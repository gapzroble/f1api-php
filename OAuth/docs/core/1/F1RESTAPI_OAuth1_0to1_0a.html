Implementer Draftnaf. OAuth Implementation Changes
 August 2009


REST API OAuth 1.0 to 1.0a (Draft)

Abstract

The API team discussed and decided to research the tasks needed to update our current OAuth lib to support version 1.0a: http://oauth.net/core/1.0a so that consumers can use the new features available in this version of OAuth.

This flow is a draft and will change over time. This flow assumes all decision points are green. Any decisions that result in rejections will loop back into the process

3.  Definitions

oauth_callback_confirmed:
MUST be present and set to true. The Consumer MAY use this to confirm that the Service Provider received the callback value.
oauth_verifier:
The verification code.
oauth_callback:
An absolute URL to which the Service Provider will redirect the User back when the Obtaining User Authorization (Obtaining User Authorization) step is completed. If the Consumer is unable to receive callbacks or a callback URL has been established via other means, the parameter value MUST be set to oob (case sensitive), to indicate an out-of-band configuration.
Fellowship One REST OAuth Definitions
All definitions from this implementation come from the [FellowshipOneOAuth] (Floyd, N., “Fellowship One OAuth Implementation,” .).

4.1.  Differences

Section 6.1.1 (Consumer Obtains a Request token)

ADDED oauth_callback

An absolute URL to which the Service Provider will redirect the User back when the Obtaining User Authorization (Obtaining User Authorization) step is completed. If the Consumer is unable to receive callbacks or a callback URL has been established via other means, the parameter value MUST be set to oob (case sensitive), to indicate an out-of-band configuration.

Section 6.1.2 (Service provider Issues an Unauthorized Request Token)

ADDED oauth_callback_confirmed

MUST be present and set to true. The Consumer MAY use this to confirm that the Service Provider received the callback value.

Section 6.2.1 Consumer Directs the User to the Service Provider

REMOVED oauth_callback

OPTIONAL. The Consumer MAY specify a URL the Service Provider will use to redirect the User back to the Consumer when Obtaining User Authorization (Obtaining User Authorization) is complete.

Section 6.2.3 Service Provider Directs the User Back to the Consumer

ADDED oauth_verifier

The verification code. If the Consumer did not provide a callback URL, the Service Provider SHOULD display the value of the verification code, and instruct the User to manually inform the Consumer that authorization is completed.

Section 6.3.1 Consumer Requests an Access Token

ADDED oauth_verifier

The verification code received from the Service Provider in the Service Provider Directs the User Back to the Consumer (Service Provider Directs the User Back to the Consumer) step.

Section 6.3.2 Service Provider Grants an Access Token

ADDED Under "The Service Provider MUST ensure that:"

The verification code received from the Consumer has been successfully verified.

5.1.  2nd Party credentials based authentication basic workflow (No Change)

1. Consumer Application collects the User's credentials directly

2. Consumer Application concatenates the user name and password with a space and base64 encodes the credentials

3. Consumer Application puts the encoded credentials in the body of the request (no parameter assignment, just put the bytes in the request)

the consumer is using the accept header value: application/x-www-form-urlencoded then the consumer must pass the credentials using the following format

ec=bXZhc3F1ZXogcGEkJHcwcmQ%3d

Credentials must be URL Encoded after they are base64 encoded

4. Consumer Application posts them to the following URI depending on what user type your using:

This request is signed using OAuth signing requests

Portal User: [POST] https://demo.fellowshiponeapi.com/v1/PortalUser/AccessToken

Weblink User: [POST] https://demo.fellowshiponeapi.com/v1/WeblinkUser/AccessToken

5. The Service Provider will hand the Consumer Application back an Access Token via:

Response body: ex. oauth_token=afd011d3-fbd3-4c69-8326-a24fad8d0c34&oauth_token_secret=ab86c226-fc65-4d32-a33c-8b54a753655e

Header:

oauth_token=afd011d3-fbd3-4c69-8326-a24fad8d0c34

oauth_token_secret=ab86c226-fc65-4d32-a33c-8b54a753655e

6. The Consumer Application will also get a link to the person via Content-Location header:

Ex. Content-Location=https://demo.fellowshiponeapi.com/v1/People/123

7. The Consumer Application will access the User's data using the Access Token and Token Secret

5.2.  3rd Party Oauth 1.0a based authentication basic workflow

1. Consumer Application requests an unauthenticated Request Token

This request is signed using OAuth signing requests

The consumer will pass in the oauth_callback parameter

[GET] https://demo.fellowshiponeapi.com/v1/Tokens/RequestToken

[POST] https://demo.fellowshiponeapi.com/v1/Tokens/RequestToken

Required header when using the [POST] verb - Content-Length: 0

2. Service Provider passes back an unauthorized Request Token

Response body: ex. oauth_token=afd011d3-fbd3-4c69-8326-a24fad8d0c34&oauth_token_secret=ab86c226-fc65-4d32-a33c-8b54a753655e&oauth_callback_confirmed=true

3. Consumer Application requests user authorization via redirect

This request is signed using OAuth signing requests

The oauth_callback parameter is not passed in here

The Consumer Application will sign the request using the Request Token and Token Secret

The Consumer Application will pass the Request Token via url

ex. https://demo.fellowshiponeapi.com/v1/PortalUser/Login?oauth_token=afd011d3-fbd3-4c69-8326-a24fad8d0c34

The Consumer Application may pass a callback url via url (optional)

ex. https://demo.fellowshiponeapi.com/v1/PortalUser/Login?oauth_token=afd011d3-fbd3-4c69-8326-a24fad8d0c34&oauth_callback=http://www.myconsumerapp.com/home

Portal User: https://demo.fellowshiponeapi.com/v1/PortalUser/Login

User logging in must be linked to a person in the Fellowship One Portal application

Weblink User: https://demo.fellowshiponeapi.com/v1/WeblinkUser/Login

4. The User enters their credentials in using the Service Provider's interface

5. The User will either Accept or Deny the request

The Service Provider will send back the oauth_verifier parameter to the Consumer Application

If the User allows access then the Service Provider authenticates the Request Token

Service Provider uses the oauth_callback parameter that was provided to the Service Provider during (Consumer Obtains a Request Token) and sends the User back to the Consumer Application

ex. Redirect http://www.myconsumerapp.com/home?oauthtoken=afd011d3-fbd3-4c69-8326-a24fad8d0c34&oauth_verifier=hfdp7dh39dks9884

If no oauth_callback parameter was provided to the Service Provider during (Consumer Obtains a Request Token) the Service Provider sends the User to another page with the authorized Request Token written in the body with the oauth_verifier parameter

If the User denies access then the Service Provider marks the Request Token as revoked

Service Provider uses the oauth_callback parameter that was provided to the Service Provider during (Consumer Obtains a Request Token) and sends the User back to the Consumer Application

ex. Redirect http://www.myconsumerapp.com/home?permissiondenied=The+user+has+denied+access+to+all+protected+resources. (OAuth problem reporting extension)

If no oauth_callback parameter was provided to the Service Provider during (Consumer Obtains a Request Token) the Service Provider sends the User to another page stating that Request Token has been revoked

6. The Consumer Application will take the Authorized Request Token and it's corresponding Token Secret and request an Access Token

This request is signed using OAuth signing requests

Consumer Application must pass in the oauth_verifier parameter with this request

The Consumer Application will sign the request using the Authenticated Request Token and Token Secret

The Consumer Application will pass the Authenticated Request Token via ur

[GET] https://demo.fellowshiponeapi.com/v1/Tokens/AccessToken

[POST] https://demo.fellowshiponeapi.com/v1/Tokens/AccessToken

Required header when using the [POST] verb - Content-Length: 0

7. The Service Provider will hand the Consumer Application back an Access Token via:

Service Provider will verify the verification code provided by the Consumer

Response body: ex. oauth_token=afd011d3-fbd3-4c69-8326-a24fad8d0c34&oauth_token_secret=ab86c226-fc65-4d32-a33c-8b54a753655e

Header:

oauth_token=afd011d3-fbd3-4c69-8326-a24fad8d0c34

oauth_token_secret=ab86c226-fc65-4d32-a33c-8b54a753655e

8. The Consumer Application will also get a link to the person via Content-Location header:

ex. Content-Location=https://demo.fellowshiponeapi.com/v1/People/123

9. The Consumer Application will access the User's data using the Access Token and Token Secret

6.  Considerations

The oauth_callback parameter will need to be stored with the request_token

The oauth_verifier value will need to be stored with the authenticated request_token

We will have to label all parameters 1.0 or 1.0a specific or inclusive

When 1.0a is used the querystring for oauth_callback will be ignored and the one stored with the request_token will be used

The oauth_verifier parameter should be unique and should be stored
